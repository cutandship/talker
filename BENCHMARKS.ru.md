# Talker — инженерные замеры

Talker — **локальный приватный голосовой ввод для Windows на русском**. Распознавание идёт
полностью на устройстве (без облака), поэтому качество — это инженерия, а не чужой API. Этот
документ фиксирует **решения по данным**, стоящие за качеством. Каждая цифра воспроизводима —
скрипты лежат в [`tests/`](tests/).

**STT-движок:** `gigaam-v3-e2e-rnnt` (Sber GigaAM, только русский, работает локально через
`onnxruntime`, **сам ставит пунктуацию и регистр**). Все замеры — на CPU.

---

## Кратко

| Вопрос | Что показал замер | Решение |
|---|---|---|
| Нужна ли модель пунктуации? | **Родная** пунктуация GigaAM = **81.7%** F1 по запятым; RUPunct поверх **роняет до 68.1%** | **Нет.** Родная лучше. |
| Здоров ли словарь брендов/терминов? | Снят на GigaAM **v2**; продакшн на **v3**, слышит иначе → **5/16** | **Пересняли** на v3 → 93 правила / 158 вариантов |
| Чинит ли словарь любой акроним? | Большинство — да; но **ИИ→«и»**, **IT→«эти»** STT уничтожает | **Нет.** Жёсткий предел — задокументирован |

Сквозная мысль: мы **меряем до того, как интегрировать**. Дважды замер развернул «очевидный» ход.

## 1. Пунктуация: реально ли нужна модель пунктуации?

### Контекст
Пунктуация — самое частое слабое место диктовок на Whisper (он режет аудио по 30 с и теряет
знаки на стыках). Обычное лечение — второй проход языковой моделью, а это ровно то, что рискует
**переписать слова пользователя**. Вопрос: нужна ли Talker модель пунктуации/очистки вообще?

### Метод
Два эксперимента, скоринг против золотого русского корпуса (Leipzig news+wiki, 12k предложений),
выравнивание по словам через `difflib` (чтобы ошибки слов STT не портили метрику пунктуации).

1. **Восстановление** ([`run_punct_bench.py`](tests/run_punct_bench.py)): срезаем пунктуацию с
   эталона → восстанавливаем. Текущая эвристика-fallback vs **RUPunct_small** (BERT
   token-classification, MIT). *10 000 предложений.*
2. **Продакшн-реальность** ([`punct_gigaam_bench.py`](tests/punct_gigaam_bench.py)): эталон →
   локальный TTS (Windows SAPI, ru-RU) → **реальная транскрипция GigaAM v3** → сравниваем его
   **родную** пунктуацию vs **переразметку тех же слов через RUPunct**. *250 предложений.*

### Результаты

**Восстановление vs голая эвристика** (10k) — тут RUPunct выглядит отлично:

| Метрика | Эвристика (текущий RU-fallback) | RUPunct_small |
|---|---|---|
| **Запятые F1** | 0.0% | **76.2%** |
| Капитализация | 92.0% | 96.8% |
| Полное совпадение | 16.2% | 37.5% |

**Но против реального движка** (250) — картина переворачивается:

| Метрика | **GigaAM-native** | GigaAM → RUPunct |
|---|---|---|
| **Запятые F1** | **81.7%** | 68.1% |
| Запятые precision | 80.9% | 74.5% |
| Запятые recall | **82.5%** | 62.6% |
| Тип концевого знака | 98.0% | 97.6% |
| Капитализация | 97.9% | 97.5% |
| Полное совпадение | 29.6% | 24.4% |

### Вывод
Первый бенчмарк **сам по себе вводит в заблуждение**: RUPunct бьёт эвристику, которая не ставит
*ни одной* внутренней запятой — но продакшн это не эвристика, а GigaAM с сильной родной
пунктуацией (81.7% F1). Переразметка выхода GigaAM через RUPunct **теряет recall и
пересегментирует** («…государства**.** право…»). Поэтому **модель пунктуации в GigaAM-путь не
тащим** — это стоило бы и качества, и зависимости `torch`/ONNX (~2 ГБ). RUPunct остаётся
доступен только для опционального Whisper-движка, где родная пунктуация слабее.

> **Здесь пунктуация — конкурентное преимущество, а не дыра**: она из on-device-модели, без
> облачного LLM-«причёсывания», которое могло бы изменить твои слова.

### Воспроизвести
```bash
python tests/punct_gigaam_bench.py --n 250   # native vs RUPunct на реальном выходе GigaAM
python tests/run_punct_bench.py --n 10000    # восстановление: эвристика vs RUPunct
```

## 2. Держим словарь терминов в синхроне с моделью STT

### Контекст
GigaAM — только русский и выдаёт кириллицу, поэтому «Claude» или «Microsoft» из речи он отдать
не может — возвращает `клод`, `майкрософт`. Talker чинит это **детерминированно после STT**
([`replacements.py`](replacements.py)): курируемая карта кириллических мисхиров → канон. Так как
это пост-STT замена (а не генеративная модель), она **не выдумывает и не переставляет слова** —
та же гарантия верности, что и во всём пайплайне.

Засада: эти мисхиры **зависят от модели**. Словарь снимали на GigaAM **v2**, а продакшн уехал
на **v3**.

### Метод
Эмпирический зонд ([`probe_gigaam.py`](tests/probe_gigaam.py)): каждый канон
([`dict_terms.py`](tests/dict_terms.py), 178 терминов) → `edge-tts` (2 голоса × 2 карьер-фразы)
→ 16 кГц моно → **GigaAM v3** → реальный кириллический мисхир. Каждый кандидат классифицируется:
- **safe** → в словарь;
- **омоним** (частотное русское слово — частотный фильтр) → **исключить** (сломает обычную речь:
  `Grok→игрок`, `Git→гид`, `ChatGPT→чат`);
- **необратимо** (схлопнут/обрезан до 1–2 букв) → отдельная корзина;
- **уже канон** (v3 сам выдал латиницу) → правило не нужно.

### Результаты
На контрольной выборке из 16 терминов **словарь v2 поймал лишь 5/16** — не из-за пробелов в
покрытии, а потому что **v3 слышит иначе**:

| Термин | Словарь v2 ждёт | Что **v3** реально выдаёт |
|---|---|---|
| Gemini | джемини / гемини | **джимней** |
| Docker | докер | **дакр** |
| Copilot | капайлот | **капайд** |
| GitHub | гитхаб | **гиток** |
| Kubernetes | кубернетис | **губернитис** |
| OpenAI | опен эй ай | **опеной** |

После пересъёма на v3 и слияния безопасных форм продакшн-словарь вырос до **93 правил /
158 вариантов**, омонимы отсечены частотным фильтром плюс ручным аудитом.

### Вывод
**Словарь замен тихо «гниёт» при апгрейде модели STT.** Считать его «настроил один раз» —
ловушка; его надо переснимать под каждую версию модели. Зонд превращает это в воспроизводимую
15-минутную задачу вместо гадания.

> **Оговорка (честно):** произношение TTS ≠ живой голос, поэтому конкретные кириллические формы —
> это *кандидаты* до сверки голосом, а не истина.

### Воспроизвести
```bash
python tests/probe_gigaam.py --full   # 178 терминов → реальные мисхиры v3 + классификация
```

## 3. Потолок акронимов

### Контекст
Акронимы по буквам (ИИ, API, URL, SQL) — известная боль STT. Прогнали, как GigaAM v3 рендерит
их в разных произношениях ([`probe_acronyms.py`](tests/probe_acronyms.py)).

### Результаты
- **Большинство в порядке.** При буквенном произношении v3 сам выдаёт правильную латиницу:
  `HTML`, `CSS`, `USB`, `CPU`, `GPU`, `PDF`, `VPN`. Правило не нужно.
- **Несколько мапятся чисто** (добавлены в словарь): `урл→URL`, `эскуэль→SQL`, `сипью→CPU`,
  `пдеф→PDF`.
- **Два необратимы** — STT уничтожает информацию:
  - **ИИ → «и»** (схлопывается в союз «и», во всех произношениях);
  - **IT → «эти»** (коллизия с частым словом «эти»).

### Вывод
**Пост-STT словарь не может вернуть то, что STT уже выбросил.** Нельзя замапить `и → ИИ` или
`эти → IT`, не сломав обычный русский. Честные лечения — выше по потоку (биасинг на уровне STT,
которого GigaAM не умеет, а Whisper умеет) или поведенческие (говорить «искусственный интеллект»,
которое распознаётся идеально). Знание, *где* потолок, экономит бесполезную работу над словарём.

### Воспроизвести
```bash
python tests/probe_acronyms.py
```

## Оговорки

- **TTS — прокси живой речи.** SAPI/edge-tts произносят пунктуацию с чистыми паузами и читают
  иностранные слова по-своему, поэтому абсолютные цифры оптимистичны относительно живых юзеров.
  *Сравнения* (native vs RUPunct; v2 vs v3) валидны, т.к. обе стороны видят одинаковый вход, но
  конкретные формы словаря всё ещё требуют **сверки голосом**.
- **Один корпус** (Leipzig RU news+wiki) — формальная проза, не чат/код-диктовка.
- Одна языковая пара (RU → канонические англ. бренды).
