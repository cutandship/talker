# 30. Screen-context через локальный OCR

> Разовый скриншот активного окна → on-device OCR → извлечённый текст подаётся как контекст в STT (`initial_prompt`) и/или в промпт очистки. Сильная реализация запланированного «контекста экрана» (концепт 25): ловит имена/термины/ники, которые видны на экране, без облака.

**Категория:** точность — усиление концепта 25.
**Эффект:** 🔥 высокая отдача (особенно имена/термины на русском).
**Готовность концепта:** 🟡 Medium.
**Источник идеи:** [research/2026-05-29](research/2026-05-29_what-to-add-next.md) F2 (VoiceInk «Context Aware»).

---

## Зачем

Концепт 25 читает текст у курсора через UIA — но это работает только в текстовых полях с UIA TextPattern, и не видит то, что на экране, но не в фокусе (ветка чата, имя собеседника, термины в соседнем сообщении). OCR-скриншот снимает это ограничение: «что я вижу» → контекст для распознавания. VoiceInk показывает, что это заметно повышает точность правок имён/терминов.

25 (UIA-каретка) и 30 (OCR) **дополняют** друг друга: UIA — быстрый путь, OCR — для приложений без UIA и для общевизуального контекста.

---

## Технический подход

### OCR на Windows без тяжёлых зависимостей

`Windows.Media.Ocr` (встроен в Windows 10/11, через WinRT-проекцию `winrt` / `winsdk`) — офлайн, поддерживает русский, не требует внешних бинарников. Альтернатива: `RapidOCR` (onnxruntime — переиспользует стек концепта 29) или Tesseract (нужен бинарник).

```python
# screenshot активного окна (по hwnd из modes.get_foreground_info)
# → OCR → текст (cap по длине)
def ocr_context(hwnd) -> str:
    img = grab_window(hwnd)            # mss / PrintWindow
    text = run_ocr(img)               # Windows.Media.Ocr
    return text[-600:]                # хвост/центр, под токен-бюджет
```

### Куда подаётся

- В `transcriber.context` (концепт 25 уже завёл этот атрибут) → `build_initial_prompt(..., context=)`.
- И/или в `extra_instructions` очистки (концепт 23) как «на экране видно: …».
- Бюджет: словарь приоритетнее, OCR-контекст добивает остаток (как в 25).

## Архитектура

- `screen_context.py` — новый: захват окна + OCR (ленивая инициализация OCR-движка).
- `cursor_format.py` — `gather_context()` (концепт 25): добавить OCR как источник после UIA-каретки.
- `config.py` — `SttConfig.context_ocr: bool = False` (по умолчанию off — privacy/скорость).
- `main.py` — `_process` уже ставит `transcriber.context` перед transcribe (концепт 25); OCR подмешивается там же.
- `ui.py` — чекбокс «OCR-контекст экрана (медленнее, точнее)».

## Зависимости

`winsdk`/`winrt` (для Windows.Media.Ocr) ИЛИ `rapidocr-onnxruntime`. `mss` для скриншота.

## Edge cases

| Сценарий | Поведение |
|---|---|
| Приватность | По умолчанию OFF; password-поля/чувствительные окна не снимаем (проверка как в injector). |
| Латентность | OCR добавляет ~100–400 мс → только когда включено; не блокировать UI. |
| Многомониторность | Снимать только активное окно по hwnd, не весь экран. |
| Мусорный OCR | Чистить, ограничивать длину; пустой → ведём себя как без контекста. |
| Не-UIA приложение | Здесь OCR особенно полезен (UIA-каретка недоступна). |

---

## Acceptance criteria

- При включённом OCR имя/термин, видимые на экране, распознаются точнее (smoke).
- Выключено по умолчанию; включение не ломает поток, лишь добавляет задержку.
- Password/чувствительные окна не снимаются.
