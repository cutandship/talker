# 31. Митинг-режим с локальной диаризацией

> Авто-детект приложений-звонков (Zoom/Teams/Meet) + живая on-device диаризация «кто говорил» поверх уже существующей записи системного звука. Превращает запись митинга в структурированные заметки со спикерами — полностью локально.

**Категория:** новый сценарий (meeting notes).
**Эффект:** 👍 средняя-высокая отдача (под спрос на митинги).
**Готовность концепта:** 🟠 Research (диаризация требует прототипа/бенчмарка).
**Источник идеи:** [research/2026-05-29](research/2026-05-29_what-to-add-next.md) F3 (OpenWhispr).

---

## Зачем

У Talker уже есть запись системного звука (WASAPI loopback, [loopback_recorder.py](../loopback_recorder.py), `audio.source = system`). Не хватает двух вещей, чтобы это стало митинг-режимом:
1. **Авто-детект**, что идёт звонок (Zoom/Teams/Meet) — чтобы не включать вручную.
2. **Диаризация** — разметка «кто говорит», иначе получается сплошная стена текста.

OpenWhispr показывает, что это делается локально: модели `pyannote-segmentation-3.0` + `CAM++` (3D-Speaker) в ONNX, отпечатки голосов в SQLite, live-метки + офлайн batch-уточнение.

---

## Технический подход

### Детект звонка

`modes.get_foreground_info()` ([modes.py:70](../modes.py)) уже умеет читать процесс окна. Расширить: следить за запущенными процессами `Zoom.exe`/`Teams.exe`/`ms-teams.exe` + (для Meet в браузере) активность микрофона. При детекте — предложить/автостартовать loopback-запись.

### Диаризация (ONNX, без PyTorch)

- `pyannote-segmentation-3.0` (VAD + сегментация перекрытий) и `CAM++`/ECAPA эмбеддер спикеров — оба портируемы в ONNX → переиспользуем onnxruntime-стек (концепт 29).
- Кластеризация эмбеддингов → метки `Спикер 1/2/…`; известные голоса — по fingerprint из SQLite.
- Схема «live + batch»: грубые метки в реальном времени, точное уточнение после завершения звонка.

### Вывод

Транскрипт с разметкой спикеров → в Историю / экспорт (у Talker уже есть SRT/VTT/JSON, [exporters.py](../exporters.py)) + опционально LLM-саммари встречи (переиспользует цепочку очистки с другим промптом, как Command Mode).

## Архитектура

- `diarization.py` — новый (ленивый onnxruntime; сегментация + эмбеддинги + кластеризация).
- `meeting.py` — новый: детект процессов-звонков, оркестрация записи+диаризации, хранение fingerprint (SQLite).
- `loopback_recorder.py` — переиспользуется как источник аудио.
- `modes.py` — детект процессов звонков.
- `exporters.py` — формат с спикерами.
- `config.py` — секция `[meeting]` (enabled, авто-старт, список процессов).
- `ui.py` — tray-пункт «Митинг-режим», окно с транскриптом по спикерам.

---

## Зависимости

`onnxruntime` + ONNX-веса pyannote-segmentation-3.0 и CAM++. SQLite (stdlib).

## Edge cases

| Сценарий | Поведение |
|---|---|
| Перекрытие речи | Диаризация слабеет — помечаем сегмент как «несколько голосов». |
| Cold-start / реплики <0.8 с | Низкая уверенность — метка «Спикер ?». |
| Приватность/согласие | Запись разговора — чувствительно; явный индикатор записи + согласие. |
| Только свой микрофон vs loopback | Поддержать оба: свой голос отдельной дорожкой. |
| Совместимость моделей с onnxruntime без PyTorch | Проверить на прототипе (открытый вопрос). |

## Acceptance criteria

- При запуске Zoom/Teams — предложение включить митинг-режим.
- Транскрипт размечен по спикерам (≥2) с приемлемой точностью на чистой речи.
- Экспорт и (опц.) LLM-саммари встречи.
- Всё локально, явный индикатор записи.
