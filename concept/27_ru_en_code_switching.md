# 27. Код-switching RU/EN — латиница для терминов

> На русско-английской технической речи Whisper транслитерирует англоязычные термины кириллицей («дип лёрнинг», «реакт», «коммит»). Фича возвращает им латинское написание, опираясь на словарь, и стабилизирует распознавание смешанной речи.

**Категория:** Tier 3 — важно для IT-аудитории (основной юзкейс — переписка с ИИ).
**Эффект:** 👍 хорошо.
**Готовность концепта:** 🟡 Medium.

---

## Зачем

Дефолт `stt.language = "ru"` ([config.py:97](../config.py)) фиксирует русский — это правильно для качества, но Whisper тогда пишет английские термины кириллицей: «закоммить в гит», «деплой на продакшн», «реакт-компонент». Для технической переписки нужно «commit в git», «deploy на prod», «React-компонент».

При `language=""` (автоопределение) Whisper, наоборот, перескакивает на английский посреди фразы и портит русскую часть. Нужна золотая середина: **русский основной + латиница для известных терминов**.

---

## Технический подход

Фича — это **специализированное применение словаря** (концепты 05/22/23), не новый STT-механизм:

1. **Bias к латинице.** Латинские термины из словаря в `initial_prompt` уже толкают Whisper писать их латиницей (концепт 05). Усиливаем: формулировка prompt с явными латинскими примерами.
2. **Пост-замена транслита → латиница.** Главный слой: для каждого латинского термина словаря держим его кириллический транслит и заменяем (через движок концепта 22, фонетика из 26): `реакт→React`, `коммит→commit`, `гит→git`, `деплой→deploy`. Это и есть «чтобы не путались».
3. **Глоссарий в очистку** (концепт 23) — LLM дочищает падежи вокруг латиницы («в React'е», «закоммить»).

```toml
[[replacement]]
to       = "React"
sounds   = "реакт"
phonetic = true
```

То есть отдельного кода фича почти не требует — это **набор готовых правил** + чуть более явный prompt. Ценность концепта — в признании юзкейса и поставке стартового набора IT-терминов.

### Поставляемый стартовый набор

Сидируем (как `default_modes_data` / `default_commands`) словарь-замен частыми транслитами: git, commit, push, pull, merge, deploy, prod, staging, React, Vue, Docker, Kubernetes/k8s, API, JSON, backend, frontend, bag→bug, фича→feature (опц.) и т.д. Пользователь правит под себя.

## Edge cases

| Сценарий | Поведение |
|---|---|
| Термин-омоним («пуш» как push vs «пушка») | `whole_word` + фонетический порог; «пуш» рискованно — в дефолтный набор не класть, или только с маркером контекста. |
| Падежи на латинском термине («в докере») | Замена даёт «в Docker», LLM-очистка (концепт 23) приводит к «в Docker». Без LLM — оставляем «Docker». |
| Юзер пишет на чистом русском | Набор правил не трогает слова, которых нет в нём. |
| Code-режим (IDE, концепт 07) | Особенно полезно — там латиница критична; mode-промпт это уже поощряет. |
| Конфликт с обычным русским словом | Решается пер-правилом и порогом фонетики (концепт 26). |

## Acceptance criteria

- «закоммить изменения в гит и задеплоить на прод» → «закоммитить изменения в git и задеплоить на prod» (с включённым набором).
- Чисто русские фразы без IT-лексики не меняются.
- Стартовый IT-набор добавляется одной кнопкой и редактируется.

## Открытые вопросы

- Насколько агрессивный дефолтный набор? → консервативный: только однозначные термины (git, React, Docker), спорные («пуш», «таск») — опционально.
- Хранить транслиты централизованно или в тех же `[[replacement]]`? → в `[[replacement]]`, единое хранилище.
