# 34. Латентность/точность «бесплатно»: спекулятивный декод + post-stream refinement

> Два приёма, повышающих скорость и точность без новых модулей: спекулятивный декод Whisper (~2x быстрее при идентичном выводе) и post-stream refinement (быстрые partials + точный финал вторым проходом по полному аудио).

**Категория:** UX потока ввода — ускорение/точность поверх существующего streaming.
**Эффект:** 👍 средняя-высокая отдача.
**Готовность концепта:** 🟡 Medium (нужен замер выигрыша на CPU).
**Источник идеи:** [research/2026-05-29](research/2026-05-29_what-to-add-next.md) F6.

---

## Зачем

У Talker уже есть streaming partial inserts (`_BgJob`, концепт 01) и full-decode на финале ([transcriber.transcribe](../transcriber.py)). Можно ускорить декод и поднять точность финала, ничего не переписывая по архитектуре.

---

## Часть A — спекулятивный декод (Whisper-ветка)

**Что.** Маленькая draft-модель (distil-whisper / whisper-tiny) предлагает токены, main-модель их проверяет батчем. Вывод **математически идентичен** main-модели → **WER не меняется** (EN 3.5→3.5, NL 12.8→12.8) при greedy/низкой температуре — это ровно режим Talker. Ускорение ~2x (EN large-v2 73→33 с; NL 117→62 с).

**Подход.** В Whisper-ветке подключить draft-модель (поддержано в HuggingFace Transformers `assistant_model`). Применимо к medium/large-v2. **Несовместимо с large-v3** (расширенный словарь). Бенчмарки опубликованы на GPU → **сначала замерить выигрыш на CPU** (faster-whisper/CTranslate2 — отдельный вопрос совместимости со спек-декодом; возможно, потребуется HF-бэкенд для этой опции).

---

## Часть B — post-stream refinement (паттерн)

**Что.** Двухпроходная схема: пользователь получает быстрые partials по ходу речи, а после завершения сегмента второй проход по **полному** аудио даёт более точный финал, которым заменяется вставленный текст. Развязывает «латентность vs точность» (Azure запустил это как фичу; берём как паттерн, не как API).

**Подход.** У Talker уже есть основа: `_BgJob` стримит partials, а на отпускании клавиши финал может полностью переdecode-иться и заменить вставленное (механика rollback через Backspace×N уже реализована, [main.py:824](../main.py) — «Stream: rolled back»). То есть Часть B — это в основном **настройка/докрутка существующего**: гарантировать, что финал всегда идёт полным декодом по полному аудио, а partials — быстрым (greedy/меньший beam). Подтверждено академически (arXiv 2312.09463).

---

## Архитектура

- `transcriber.py` — опция `assistant_model` для спек-декода (Часть A); параметры быстрых partials vs точного финала (Часть B).
- `main.py` — `_BgJob`/`_process` уже содержат streaming + rollback (концепт 01); докрутить разделение «быстрый partial / точный финал».
- `config.py` — `[stt]`: `speculative: bool`, `draft_model: str`; (опц.) параметры partial-декода.
- `requirements.txt` — при HF-пути: `transformers`, draft-модель.

---

## Зависимости

Часть A: draft-модель (distil-whisper/tiny) + бэкенд с поддержкой спек-декода. Часть B: нет (докрутка существующего).

---

## Edge cases

| Сценарий | Поведение |
|---|---|
| Модель large-v3 | Спек-декод несовместим → опция недоступна, понятное сообщение. |
| Выигрыш на CPU не подтверждён | Сначала бенчмарк; если на CPU мало — оставить выключенным по умолчанию. |
| Финал отличается от partials | Rollback (Backspace×N) + вставка точного финала — уже есть. |
| Parakeet/GigaAM | Спек-декод — только Whisper; для них Часть A не применяется. |
| Короткие реплики (≤20 с) | Финал и так full-decode ([main.py:793](../main.py)); выигрыш в основном на длинных. |

---

## Acceptance criteria

- (A) При включённом спек-декоде декод заметно быстрее на CPU при том же тексте (smoke + замер RTF); иначе — оставить off.
- (B) Partials появляются быстро, финал — точный (full-decode по полному аудио), замена без визуальных артефактов.

---

## Сложность

- Часть A: ~4–6 часов + бенчмарк. Часть B: ~3–4 часа (докрутка). Делать A только если CPU-выигрыш подтверждён.

---

## Открытые вопросы

- Даёт ли спекулятивный декод выигрыш на **CPU** (бенчмарки были на GPU T4)?
- Совместим ли спек-декод с faster-whisper/CTranslate2, или нужен HF-бэкенд (тяжелее)?
- Оптимальные параметры быстрых partials (greedy, beam=1) без заметной потери качества промежуточного текста.
