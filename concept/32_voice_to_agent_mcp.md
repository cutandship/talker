# 32. Voice-to-agent: диктовка → действия через MCP

> Расширить Command Mode: голос → planner-LLM → вызов инструментов через MCP → результат. Не только править текст голосом, но и запускать действия (поиск, файлы, API), оставаясь в основном flow «голос → текст в окно».

**Категория:** новый сценарий (агентность поверх диктовки).
**Эффект:** 👍 средняя-высокая отдача.
**Готовность концепта:** 🟠 Research (латентность + безопасность инструментов).
**Источник идеи:** [research/2026-05-29](research/2026-05-29_what-to-add-next.md) F4 (OpenAI MCP cookbook, OpenWhispr).

---

## Зачем

У Talker уже есть Command Mode (голосом править выделенный текст через LLM, [main.py:_process_command](../main.py)) и inline-команды. Логичный следующий шаг — дать голосу запускать **действия**, а не только редактировать текст: «найди в почте письмо от Иванова», «создай заметку», «открой PR». Референс-архитектура (OpenAI Cookbook): голос → STT → planner-LLM → инструменты через **MCP** → (опц.) ответ. OpenWhispr уже делает это на практике, различая «команда агенту» и обычную диктовку.

---

## Технический подход

### MCP-клиент поверх существующей LLM-цепочки

- Тот же стек, что в очистке ([cleaner.py](../cleaner.py)): OpenRouter/Ollama/локальный LLM как planner. Добавить MCP-клиент, который подключает сервера инструментов (modelcontextprotocol) и даёт planner'у tool-calling.
- Планировщик решает: это диктовка (вставить текст) или команда агенту (вызвать инструменты). Детект — по marker-слову (как inline voice commands, [voice_commands.py](../voice_commands.py)) или отдельному хоткею (как Command Mode).
- Результат: вставить ответ в окно, либо выполнить действие и показать статус в bubble.

```
голос → STT → planner-LLM (tool-calling) → MCP tools → результат → вставка/статус
```

### Режимы

- Отдельный хоткей «агент» (рядом с Command Mode hotkey, `command_mode.hotkey`).
- Marker-слово в обычной диктовке («talker, найди …»), вырезается из текста (механика уже есть в `extract_commands`).

## Архитектура

- `agent_mode.py` — новый: MCP-клиент, planner-цикл (tool-calling), безопасность.
- `cleaner.py` / LLM-слой — переиспользовать бэкенды как planner (нужна модель с function/tool-calling).
- `main.py` — новый обработчик `_on_agent_press`/`_process_agent` по образцу `_process_command` ([main.py:1083](../main.py)).
- `config.py` — секция `[agent]` (enabled, hotkey, список MCP-серверов, allow-list инструментов).
- `ui.py` — настройка MCP-серверов и разрешений.

---

## Зависимости

MCP-клиент (SDK `mcp`) + LLM с tool-calling. Конкретные MCP-серверы — по выбору пользователя.

---

## Edge cases

| Сценарий | Поведение |
|---|---|
| Латентность | Цепочка добавляет ~0.8–2 с; показывать «думаю…» в виджете. |
| Безопасность инструментов | MCP «безопасность» — цель, не гарантия (tool-poisoning/injection). Нужны свой allow-list, подтверждение деструктивных действий. |
| Нет tool-calling у модели | Fallback: обычная очистка/диктовка; предупредить. |
| Двусмысленность «команда vs диктовка» | Дефолт — marker/хоткей (как inline-команды), а не угадывание. |
| Деструктивные действия (удаление/отправка) | Запрашивать подтверждение, не выполнять молча. |

---

## Acceptance criteria

- По хоткею/маркеру голосовая команда вызывает MCP-инструмент и возвращает результат.
- Обычная диктовка по-прежнему просто вставляет текст (без регрессий).
- Деструктивные действия требуют подтверждения; allow-list соблюдается.
