# Концепция: голосовая надиктовка для общения с ИИ (Windows)

> Документ-задание для Claude Code. Описывает, что строим, из чего и в каком порядке.
> Реализация — Python, под Windows, без видеокарты, с прицелом на минимальный и понятный код.

---

## 1. Что это и зачем

Небольшая фоновая программа для Windows: пользователь зажимает горячую клавишу,
говорит вслух, отпускает — и в активное окно вставляется **чистый, отредактированный текст**.

Это не диктовка кода. Это инструмент, чтобы голосом вести переписку и обсуждения
с ИИ-ассистентами (и вообще писать прозу в любое поле ввода), не печатая руками.

Аналог по идее — Wispr Flow, но: своё, локальное, открытое, минимальное, под себя.

Ключевая мысль: программа состоит из **двух моделей**, и обе важны.
1. **Распознавание речи** (STT) — переводит звук в сырой текст.
2. **Очистка текста** — убирает слова-паразиты, междометия, повторы, оговорки;
   расставляет пунктуацию и абзацы. Смысл при этом не меняется.

Сырой выход STT — это дословная расшифровка устной речи со всеми «эээ», «ну вот»,
«короче» и брошенными на середине фразами. Читать и отправлять такое неприятно.
Шаг очистки — это то, что превращает бормотание в нормальный текст. Он обязателен.

---

## 2. Сценарий использования

1. Программа запущена и висит в системном трее.
2. Пользователь в любом приложении (браузер, чат с ИИ, мессенджер, редактор).
3. Зажимает горячую клавишу, говорит реплику (от пары слов до нескольких абзацев),
   отпускает клавишу.
4. Через несколько секунд в активное поле ввода вставляется готовый чистый текст.
5. Пользователь его при желании правит и отправляет.

Задержка в несколько секунд — это нормально и ожидаемо. Мгновенная субтитровка не нужна.

## 3. Чего НЕ делаем (non-goals)

Чтобы не раздувать проект, явно за рамками первой версии:

- Диктовка исходного кода (это отдельная сложная задача, не для general STT).
- Транскрипция в реальном времени с показом текста по ходу речи.
- Распознавание говорящих (диаризация), таймстампы по словам.
- Дообучение / fine-tune самих моделей. Берём готовые модели как есть.
- Кроссплатформенность. Цель — только Windows.
- GUI-настройки. Конфигурация — через текстовый файл.
- Облачное хранение, аккаунты, синхронизация.

## 4. Архитектура (пайплайн)

Один процесс, простой линейный конвейер:

```
[горячая клавиша зажата]
        │
        ▼
  захват звука с микрофона  (16 кГц, моно, в память)
        │
   [клавиша отпущена]
        │
        ▼
  STT: faster-whisper  →  сырой текст
        │
        ▼
  очистка: Cleaner     →  чистый текст
        │
        ▼
  текст в буфер обмена  →  эмуляция Ctrl+V в активное окно
```

Модель STT загружается **один раз при старте** и держится в памяти —
иначе первое распознавание будет очень долгим.

---

## 5. Технологический стек

Язык: Python 3.11 (или новее).

| Назначение | Библиотека | Примечание |
|---|---|---|
| Распознавание речи | `faster-whisper` | CTranslate2, быстро на CPU |
| Захват звука с микрофона | `sounddevice` | пишем в numpy-массив, 16 кГц моно |
| Глобальная горячая клавиша | `keyboard` | хорошо работает на Windows |
| Буфер обмена | `pyperclip` | положить текст в clipboard |
| Эмуляция Ctrl+V | `keyboard` | вставка в активное окно |
| Иконка в трее + меню | `pystray` + `Pillow` | фоновый режим |
| HTTP-запросы (очистка) | `httpx` или `requests` | к Ollama / API |
| Конфигурация | `tomllib` (встроен) + TOML-файл | читаем настройки |

GPU не используется. Все вычисления — на CPU.

## 6. Функциональные требования

### 6.1 Распознавание речи

- Движок: `faster-whisper`.
- Модель: по умолчанию `small`, размер задаётся в конфиге (`tiny` / `base` / `small`).
  Для русского `small` заметно лучше `base`; `tiny` — только если совсем тормозит.
- Брать **мультиязычную** модель (не `.en`-вариант).
- `compute_type = "int8"`, `device = "cpu"`.
- Язык не фиксировать жёстко — Whisper определяет его сам и поддерживает
  около сотни языков. В конфиге дать опциональный параметр `language`
  (если задан — форсим, если пусто — автоопределение).
- Модель грузится при старте приложения (с уведомлением в трее «загрузка…»).

### 6.2 Захват звука

- Запись стартует по нажатию горячей клавиши, останавливается по отпусканию
  (режим push-to-talk).
- Формат: 16 кГц, моно, в оперативную память (без записи на диск).
- Очень короткие записи (< ~0.3 сек) игнорировать — это случайные нажатия.
- Опционально обрезать тишину по краям (встроенный VAD faster-whisper). Не критично.

### 6.3 Горячие клавиши

- Основная клавиша push-to-talk — **настраивается в конфиге**.
  Дефолт: одна удобная для удержания клавиша (например `right ctrl` или `F9`).
- Поддержать два режима, выбор в конфиге:
  - `hold` (push-to-talk): запись, пока клавиша зажата. **Дефолт.**
  - `toggle`: первое нажатие — старт, второе — стоп.
- Отдельная команда (хоткей или пункт меню трея) — **быстрое вкл/выкл очистки**,
  чтобы можно было разово получить сырой текст.

### 6.4 Вставка результата

- Готовый текст кладётся в буфер обмена и вставляется эмуляцией `Ctrl+V`.
- Именно через буфер, **не** посимвольной эмуляцией набора — так надёжнее
  во всех приложениях и корректно с не-латиницей.
- Опционально (конфиг): сохранять прежнее содержимое буфера и возвращать его
  через секунду после вставки.

### 6.5 Фоновый режим (трей)

- Программа живёт в системном трее, окон не открывает.
- Иконка отражает состояние: ожидание / запись / обработка / ошибка.
- Меню трея: вкл-выкл очистки, перезагрузить конфиг, открыть лог, выход.
- Лёгкая звуковая или визуальная индикация старта и конца записи (по желанию).

## 7. Модуль очистки текста (ключевая часть)

Очистка — это не один бэкенд, а **цепочка из нескольких бэкендов с перебором
по приоритету**. Программа пробует их сверху вниз и берёт первый, который ответил.
Так пользователь не зависит от одного провайдера: упёрся в дневной лимит
бесплатной модели — программа молча уходит на следующую в списке.

### 7.1 Общий интерфейс

Абстрактный класс `Cleaner` с методом `clean(text: str) -> str`.
Реализации (отдельные бэкенды):

- `ApiCleaner` — OpenAI-совместимый endpoint (OpenRouter, DeepSeek и т.п.).
  Конфигурируется `base_url`, `api_key`, `model`. Требует интернет.
- `OllamaCleaner` — локальный Ollama (`http://localhost:11434`).
  Полностью офлайн, бесплатно. Скорость и качество зависят от модели.
- `NoopCleaner` — ничего не делает, возвращает текст как есть (режим «сырой текст»).

В конфиге задаётся **список** бэкендов по приоритету. Каждый элемент — это
один настроенный бэкенд (тип + его параметры). Несколько `ApiCleaner` с разными
моделями — это нормально и ожидаемо (бесплатная модель, под ней ещё одна,
внизу платная).

### 7.2 Логика перебора (цепочка)

Отдельный класс — например `CleanerChain` — принимает список бэкендов и метод
`clean(text)` работает так:

1. Берём первый бэкенд из списка, вызываем его `clean()`.
2. Если он вернул результат — отдаём результат, дальше не идём.
3. Если он упал с **повторяемой** ошибкой (`429` / лимит, таймаут, сетевой сбой,
   провайдер недоступен) — тихо переходим к следующему бэкенду в списке.
4. Если список кончился, а результата нет — это уже общий отказ очистки,
   срабатывает фолбэк на сырой текст (см. 7.5).

Важные детали:

- Перебор должен быть **незаметным для пользователя** — никаких ошибок на экран,
  просто текст приходит чуть позже. Переходы между бэкендами писать в лог.
- Различать ошибки. `429`, таймаут, сетевой сбой — повод идти дальше по списку.
  А вот `401` (неверный ключ) или `400` (кривой запрос) — это ошибка конфигурации,
  перебор её не лечит; такую залогировать заметно и идти дальше как fallback.
- Это **простой перебор списка**, а не «умный роутер». Никакого мониторинга цен,
  замеров скорости, выбора «оптимальной» модели — это переусложнение без выгоды.
  Фиксированный список по приоритету, сверху вниз.
- Опциональная мелочь: если бэкенд только что выдал `429`, можно на ближайшие
  N минут пропускать его сразу, не тратя время на заведомо неудачный запрос.
  Не обязательно для первой версии.

### 7.3 Рекомендуемый список по умолчанию

В дефолтном конфиге — цепочка такого вида (сверху вниз):

1. **Бесплатная модель** через OpenRouter (например, бесплатная Llama или Gemma).
   Ноль затрат, но есть дневной лимит запросов — поэтому она первая, но не единственная.
2. **Платная дешёвая модель** как гарантированный запасной вариант
   (например, Gemini Flash-Lite или DeepSeek — стоит центы).
   Срабатывает, когда бесплатная упёрлась в лимит.
3. (Опционально) **локальный Ollama** — если у пользователя он установлен,
   как полностью офлайновый запасной вариант на случай отсутствия интернета.

Любой пункт пользователь может убрать или переставить в конфиге.
Если интернета нет вообще и Ollama не стоит — отработает фолбэк на сырой текст.

### 7.4 Замечания по моделям

- Очистка текста — лёгкая операция (сотни токенов на запрос), стоит ничтожно мало;
  даже при активном использовании это порядка доллара в месяц, а на бесплатном
  бэкенде — ноль.
- Для очистки **не использовать reasoning-модели** (рассуждающие): они тратят
  лишние «думающие» токены и время впустую. Нужна обычная небольшая модель.
- Качество русского у мелких моделей разное. Стоит вживую сравнить пару моделей
  на своих репликах и поставить лучшую первой в списке.
- Для локального Ollama — модель класса 2–4B с хорошим русским; совсем крошечные
  склонны менять смысл и додумывать, а нужна бережная редактура.

### 7.5 Промпт очистки

Один и тот же системный промпт для локальной и облачной модели.
Примерная формулировка (можно доработать):

```
Ты — редактор расшифровок устной речи. На вход поступает дословная
транскрипция того, что человек сказал вслух. Твоя задача — отредактировать её.

Сделай:
- убери слова-паразиты и междометия (ну, эээ, как бы, короче, типа, вот);
- убери повторы и оговорки, оставь итоговый вариант фразы;
- расставь корректную пунктуацию;
- разбей текст на абзацы по смыслу.

НЕ делай:
- не меняй смысл и не перефразируй без необходимости;
- не добавляй ничего от себя, не отвечай на содержание текста;
- не переводи на другой язык, сохраняй язык оригинала;
- не добавляй заголовков, пояснений или комментариев.

Верни ТОЛЬКО отредактированный текст, без кавычек и преамбулы.
```

Текст пользователя передаётся как обычное сообщение. Низкая температура.

### 7.5 Отказоустойчивость (важно)

- Если очистка не сработала (нет интернета, Ollama не запущен, таймаут, ошибка) —
  программа **не должна падать и не должна терять текст**.
- В этом случае: вставить **сырой текст** из STT и показать в трее уведомление
  «очистка недоступна, вставлен сырой текст».
- Лучше неидеальный текст, чем потерянная реплика.
- Разумный таймаут на запрос очистки (например, 20–30 секунд).

## 8. Конфигурация

Один файл `config.toml` рядом с программой. Перечитывается из меню трея.

Примерный набор параметров:

```toml
[hotkey]
key  = "right ctrl"      # клавиша push-to-talk
mode = "hold"            # "hold" или "toggle"

[stt]
model    = "small"       # tiny | base | small
language = ""            # пусто = автоопределение, иначе код языка ("ru", "en")

[output]
restore_clipboard = true # вернуть прежнее содержимое буфера после вставки

# --- Цепочка очистки ---
# Список [[cleaner]] задаёт бэкенды по приоритету: сверху вниз.
# Программа берёт первый, который ответил. Упёрся в лимит / нет сети — идёт дальше.
# type: "api" | "ollama" | "noop"

# 1. Бесплатная модель через OpenRouter (есть дневной лимит запросов)
[[cleaner]]
type     = "api"
base_url = "https://openrouter.ai/api/v1"
api_key  = "..."
model    = "..."         # имя бесплатной модели в OpenRouter

# 2. Платный дешёвый запасной вариант (срабатывает при лимите выше)
[[cleaner]]
type     = "api"
base_url = "https://openrouter.ai/api/v1"
api_key  = "..."
model    = "..."         # дешёвая платная модель (Gemini Flash-Lite / DeepSeek)

# 3. Локальный офлайн-вариант на случай отсутствия интернета (опционально)
[[cleaner]]
type  = "ollama"
url   = "http://localhost:11434"
model = "gemma2:2b"      # имя модели в Ollama

# Если очистка нужна полностью отключённой — оставить один бэкенд type = "noop".
```

При первом запуске, если файла нет — создать его со значениями по умолчанию.
Список `[[cleaner]]` может содержать сколько угодно пунктов; пользователь
переставляет и удаляет их под себя. Перебор всегда идёт сверху вниз.

## 9. Обработка ошибок и крайние случаи

- Микрофон недоступен / занят — уведомление в трее, не падать.
- Слишком короткая запись — тихо игнорировать.
- Пустой результат STT (тишина, шум) — ничего не вставлять, уведомление.
- Очистка недоступна — фолбэк на сырой текст (см. 7.5).
- Ollama не запущен — понятное сообщение, фолбэк.
- Все нетривиальные ошибки писать в лог-файл; пункт меню «открыть лог».
- Долгая загрузка модели на старте — индикация «загрузка» в трее,
  горячая клавиша игнорируется, пока модель не готова.

## 10. План разработки по этапам

Объём небольшой; ориентир — несколько вечеров на ядро, остальное на полировку.

**Этап 1 — голый конвейер.**
Захват звука по хоткею → faster-whisper → вывод сырого текста в консоль.
Без трея, без очистки. Цель — убедиться, что распознавание работает и по скорости устраивает.

**Этап 2 — вставка и фон.**
Вставка результата в активное окно через буфер обмена. Режим push-to-talk.
Иконка в трее, работа в фоне, меню с выходом.

**Этап 3 — очистка.**
Интерфейс `Cleaner` и три реализации (`ApiCleaner`, `OllamaCleaner`, `NoopCleaner`).
Класс `CleanerChain` — перебор бэкендов из конфига сверху вниз с обработкой
повторяемых ошибок (`429`, таймаут, сеть). Фолбэк на сырой текст, если вся
цепочка не ответила.

**Этап 4 — конфиг и полировка.**
Файл `config.toml`, перечитывание из трея. Индикация состояний, звук старт/стоп,
обработка крайних случаев, лог. Подгонка хоткеев и поведения под себя.

## 11. Структура проекта

Ориентировочная раскладка файлов (держать просто и плоско):

```
voice-dictation/
├── main.py            # точка входа, цикл приложения, трей
├── recorder.py        # захват звука по хоткею
├── transcriber.py     # обёртка над faster-whisper
├── cleaner.py         # Cleaner (Api/Ollama/Noop) + CleanerChain (перебор)
├── config.py          # чтение и дефолты config.toml
├── config.toml        # настройки пользователя
├── requirements.txt
└── README.md
```

Объём кода ядра ожидается небольшим — порядка пары сотен строк.
Приоритеты: простота, читаемость, никаких лишних абстракций.
